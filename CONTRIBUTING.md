# Contributing to semantic-autoencoder

This repository is the **generic framework** for semantic encoding/decoding —
agents, skills, rules, and documentation that any project can reuse. It is a GitHub
**template repository**: each project is created from it with one click (*Use this
template*), as an independent repo with **no shared git history**. Because there is no
common ancestor, `git pull`/`git merge` cannot move improvements between the framework
and the projects — the two flows below do that instead.

---

## Two flows

**Update flow — framework → projects.** Run [`sync-framework.sh`](sync-framework.sh) from
a project. It clones the latest `semantic-autoencoder`, diffs each framework file against
the project's copy, and shows the changes for you to **review and apply selectively**. One
command, explicit review, no automatic merges:

```bash
./sync-framework.sh                # interactive, file-by-file review
./sync-framework.sh --patch-only   # write framework-sync.patch to apply by hand (hunk-level)
./sync-framework.sh --list         # list the tracked framework files
```

Each project pulls updates at its own pace; nothing is forced.

**Contribution flow — project → framework.** Spot an improvement to a generic file in your
copy? Open a PR to `semantic-autoencoder` against that file (see the PR flow below). Once
it's reviewed and merged, every project can pick it up via `sync-framework.sh`.

---

## What belongs here vs. in your project

The authoritative list of upstream-owned files is [`framework-files.txt`](framework-files.txt)
— that is what `sync-framework.sh` reads. In the markered subset (agents, skills, rules,
`CLAUDE.md`), each file also carries the header

```
<!-- FRAMEWORK FILE: improvements → PR to semantic-autoencoder -->
```

as a human-facing reminder; the manifest is the machine-facing source of truth.

**Framework (PR these back, and they sync down):** the six `.claude/agents/*.md`, the three
framework skills `.claude/skills/{setup,orchestrate,draw-workflow}/`, `.claude/rules/*.md`,
`CLAUDE.md`, `docs/workflow.md`, `docs/assets/`, `config/project_config.example.yaml`,
`CONTRIBUTING.md`, `framework-files.txt`, and `sync-framework.sh` itself.

**Project-owned (never synced down — they stay yours):** `README.md` (each project writes
its own), `.claude/settings.json` (per-project permissions), `config/project_config.yaml`
(generated by `/setup`), the domain skills `explain-domain` / `run-legacy` / `run-decoded`,
and everything under `encoded/`, `semantic_ir/`, `decoded/`, and `regression_tests/`.

> Note: the framework repo's own `README.md` *is* framework-level — fix it with a PR here —
> but it is intentionally absent from `framework-files.txt` so a sync never overwrites a
> project's own README.

## Pull-request flow

1. Branch from `main` (`simone/short-description`).
2. Make the change against the framework file(s) only — keep domain-specific edits out.
3. If you add or remove a framework file, update `framework-files.txt` in the same PR.
4. If you change the agent roster or pipeline order, regenerate the diagram with
   `/draw-workflow` so `docs/workflow.md` stays in sync.
5. Keep the PR small and single-purpose; describe what changed and why.
6. Open the PR against this repository's `main`.

---

## Conventions

- Follow `.claude/rules/framework-conventions.md` for agent/skill/rule frontmatter
  and length limits.
- Conventional-commit messages: `type: brief summary`.
- Don't introduce domain-specific assumptions into generic framework files.
